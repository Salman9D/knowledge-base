# LEO (Embodied Generalist Agent)

**Summary**: Huang et al. (Zhejiang/Shanghai AI Lab, 2023) present LEO, the first embodied generalist AI agent for 3D worlds — a single Vicuna-7B model capable of understanding 3D scenes and executing diverse tasks including captioning, QA, navigation, and robotic manipulation, all in an ego-centric streaming setting.

**Sources**: `raw/papers/An Embodied Generalist Agent in 3D World.pdf`

**Last updated**: 2026-04-27

---

## Overview

LEO (an acronym for **L**iving **E**mbodied **O**bject) is the most complete prior embodied generalist agent in the [[ego-centric-3d-perception]] paradigm. Where [[3d-llm]] requires a pre-built 3D scan and [[spatialvlm]] handles only 2D spatial QA, LEO combines:

1. Ego-centric 3D perception from streaming first-person views
2. A generalist language model backbone (Vicuna-7B)
3. Task execution across scene captioning, QA, task planning, embodied navigation, and robotic arm manipulation

All from a single model, with a single training run.

## Architecture

### Two-Stage Training

**Stage 1: Object-Level Embodied Alignment**
Train the model to understand individual 3D objects from ego-centric views.
- Input: ego-centric image crops + object-level descriptions
- Output: 3D object understanding (what it is, where it is, properties)
- Purpose: establishes grounding between visual features and language

**Stage 2: Scene-Level Embodied Instruction Tuning**
Train the model on full scene tasks with natural language instructions.
- Input: ego-centric scene video + instruction prompt
- Output: task-appropriate response (answer, plan, action sequence, or navigation waypoint)
- Fine-tuning: LoRA (Low-Rank Adaptation) on Vicuna-7B — efficient parameter-efficient fine-tuning

### 3D Scene Encoding

LEO encodes 3D scenes using a **viewpoint-dependent perceiver** that processes:
- Ego-centric RGB frames (not a pre-built point cloud)
- Depth cues from monocular estimation or stereo
- Object region features extracted from 2D detectors

The perceiver compresses the scene into a fixed-length token sequence that is concatenated with the instruction prompt before feeding into Vicuna-7B.

### Object-Grounded Chain-of-Thought (O-CoT)

A key data generation innovation: **O-CoT** creates training examples where the model's reasoning chain explicitly references 3D spatial positions.

Standard CoT: "The mug is on the shelf. The shelf is to the right. Therefore go right."
O-CoT: "The mug [position: (1.2, 0.4, 0.8)] is on the shelf [position: (1.0, 0.5, 0.7)]. The shelf is to my right (allocentric: +X). Navigation action: turn_right(45°), move_forward(1.2m)."

O-CoT data is automatically generated by augmenting existing embodied task datasets with extracted 3D positions, then prompting GPT-3.5 to write spatially-grounded reasoning chains. This creates a training signal that teaches the model to link language reasoning to explicit 3D positions.

## Task Coverage

| Task | Method | Example |
|------|--------|---------|
| 3D Scene Captioning | Describe visible scene | "A kitchen with a wooden table, four chairs, and a window on the left" |
| 3D QA | Answer spatial questions | "Where is the red mug?" → "On the shelf, 1.2m to your right" |
| Embodied Navigation | Plan + execute movement | "Go to the bedroom" → sequence of action primitives |
| Task Planning | Multi-step instructions | "Make coffee" → [open cabinet, take mug, ...] |
| Robotic Manipulation | Arm control | "Pick up the pen" → grasp trajectory |

LEO is evaluated on: ScanQA, SQA3D, ScanRefer, Multi3DRefer, 3D-VisTA, and embodied simulation environments (iGibson, VirtualHome).

## Key Results

LEO achieves state-of-the-art (at publication, late 2023) on:
- ScanQA 3D Question Answering
- SQA3D Situational QA
- ScanRefer 3D object localization

The multi-task training actually **improves performance on individual tasks** compared to task-specific models — evidence of genuine generalization from shared 3D understanding.

## Comparison with WorldGPT

Both LEO and [[worldgpt]] are built on 7B-class language models with multimodal inputs, but they address different problems:

| | LEO | [[worldgpt]] |
|--|-----|------|
| **Primary goal** | Embodied agent (perceive + act) | World model (predict state transitions) |
| **Scene input** | Ego-centric 3D (streaming) | Any modality (video/image/audio/text) |
| **Output** | Actions in physical 3D space | Next-state predictions across modalities |
| **Training** | Two-stage embodied alignment | Progressive state transition training |
| **Spatial grounding** | Explicit 3D positions | Implicit via video prediction |

LEO solves the "how does the agent act in the world?" problem. WorldGPT solves the "how does the world change?" problem. A complete spatial AI system needs both.

## Strategic Implications

LEO's design directly informs the [[spatial-ai-assistant-architecture]]:

1. **Single generalist model > task-specific ensemble**: LEO demonstrates that a single LLM fine-tuned on embodied tasks outperforms per-task specialists. For 9D Technologies, this argues for a unified model approach rather than stitching together separate scene-understanding + QA + navigation models.

2. **LoRA fine-tuning is sufficient**: Parameter-efficient fine-tuning on Vicuna-7B achieves strong results without full retraining. This is a practical deployment path for customizing to specific verticals (healthcare, training).

3. **O-CoT data generation is replicable**: The O-CoT technique (augment existing data with 3D positions, generate spatially-grounded reasoning chains via GPT) can be applied to any domain with 3D scene data. See [[synthetic-training-data]] for how this fits the broader data generation strategy.

4. **Gap to production**: LEO operates in simulation and pre-scanned environments — not live streaming from an XR headset. The gap from benchmark performance to real-time on-device inference on Quest 3 or smart glasses is significant. [[embodiedscan]] provides the next-generation training dataset for this gap.
